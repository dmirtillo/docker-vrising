## Context

The `scripts/test-startup.sh` script mounts a temporary directory (`$TEST_DIR`) from the host into the Docker container at `/mnt/vrising`. Inside the container, an initialization script runs as root, setting permissions, and then uses `gosu` to switch to the `steam` user before launching SteamCMD and the game server.

Consequently, all files downloaded and generated by SteamCMD and the game server within `/mnt/vrising` are owned by the `steam` user (or `root` during initial setup) from the container's perspective. When the host machine running the script (which is usually a standard user like `runner` in GitHub Actions or a local developer account) attempts to clean up `$TEST_DIR` using `rm -rf`, it receives a "Permission denied" error because the host user does not have ownership or write access to those container-generated files.

## Goals / Non-Goals

**Goals:**
- Ensure `scripts/test-startup.sh` cleanly deletes the `$TEST_DIR` and all its contents upon script exit (both success and failure).
- Bypass host user permission restrictions when deleting files created by the container's internal users.

**Non-Goals:**
- Changing the permission model of the actual V Rising Docker image (it should continue to run as the `steam` user for security).
- Requiring the user to run the test script with `sudo`.

## Decisions

- **Cleanup Strategy**: Modify the `cleanup` trap function in `scripts/test-startup.sh`. Before attempting to remove the directory on the host, the script will spawn a short-lived Docker container (`ubuntu:24.04`) with the `$TEST_DIR` mounted. This temporary container will run a `bash` command to delete all contents (`*` and `.*`) of the mounted directory as the `root` user.
  - *Alternative Considered*: Running `sudo rm -rf` on the host. This was rejected because `sudo` may require a password prompt for local developers, breaking the automated nature of the script.
  - *Alternative Considered*: Trying to run an `rm` command inside the primary test container before stopping it. This was rejected because the test container might have crashed or entered a bad state, making cleanup unreliable. A fresh, dedicated cleanup container is more robust.

## Risks / Trade-offs

- **Performance**: Spinning up a second container just for cleanup adds a couple of seconds to the script execution time. This is a negligible trade-off for guaranteed cleanup reliability.
- **Docker Dependency**: The cleanup now strictly requires Docker to be running and responsive. Since the script's core purpose is to test a Docker image, this is an acceptable assumption.
